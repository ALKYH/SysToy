# SysToy Phase 6 FAT32 说明

## 1. 范围

本文档覆盖工作包 `6.1` 到 `6.6`，其中当前实现重点落在：

1. FAT32 基础结构调研与建模
2. 虚拟盘创建与格式化流程定义
3. 文件系统信息展示能力
4. 将 ELF 写入虚拟盘
5. 内核从 FAT32 中读取并装载 ELF
6. 文件系统边界与扩展说明

## 2. 工作包 6.1：FAT32 基础结构调研与建模

当前最小支持范围如下：

1. 只讨论 FAT32 的基础磁盘布局。
2. 重点关注 boot sector、FAT 区、数据区、目录项。
3. 不覆盖长文件名、碎片整理、坏簇恢复、权限模型等高级能力。

当前要求团队至少能说明：

1. boot sector 中哪些字段决定文件系统几何布局。
2. FAT 区的数量、位置和基本职责。
3. 根目录从哪个 cluster 开始。
4. 一个 32 字节目录项如何表示一个短文件名文件。

## 3. 工作包 6.2：虚拟盘创建与格式化流程定义

当前仓库给出的最小流程是：

1. 创建一个固定大小的磁盘镜像文件。
2. 写入一份最小 FAT32 boot sector。
3. 初始化 FAT1 和 FAT2。
4. 为根目录 cluster 预留数据区空间。

当前最小约定：

1. `bytes_per_sector = 512`
2. `sectors_per_cluster = 1`
3. `reserved_sector_count = 32`
4. `fat_count = 2`
5. `root_cluster = 2`

这套参数不是为了兼容所有真实 FAT32 工具，而是为了提供一个教学上可解释、可解析、可验证的最小镜像。

## 4. 工作包 6.3：文件系统信息展示能力

当前目标不是“实现完整 FAT32 驱动”，而是“把镜像结构解释出来”。

最小展示能力包括：

1. 读取 boot sector 并显示关键字段。
2. 计算 FAT 起始扇区与数据区起始扇区。
3. 说明 root cluster 的位置。
4. 输出结构化解析结果，便于后续脚本或自动测试使用。

## 5. 当前边界

当前仍未覆盖：

1. 多级目录遍历
2. FAT 链跨多个 cluster 的复杂文件
3. 长文件名
4. 删除、重命名、追加写入
5. 内核态块设备驱动

## 6. 当前交付物

与 6.1 到 6.6 对应的当前交付物有：

1. `tools/make_fat32_image.py`
2. `tools/show_fat32_layout.py`
3. FAT32 根目录中的 `TASKA.ELF` 与 `TASKB.ELF`
4. 内核中的最小 FAT32 root 目录读取逻辑
5. 本说明文档

## 7. 工作包 6.4：ELF 写入虚拟盘

当前方案采用最小 8.3 文件名目录项，把以下文件写入 root directory：

1. `TASKA.ELF`
2. `TASKB.ELF`

写入方式是：

1. 在工具脚本中读取 `build/user_a.elf` 与 `build/user_b.elf`
2. 生成 root directory entry
3. 将文件内容写入 cluster 3 和 cluster 4
4. 在 FAT1/FAT2 中将这些 cluster 标记为文件终止

## 8. 工作包 6.5：内核从 FAT32 读取并装载 ELF

当前最小实现采用“镜像二进制嵌入内核，再在内存中解析 FAT32”的方式，而不是直接实现真实磁盘 I/O 驱动。

这样做的原因是：

1. 保持和当前教学内核复杂度一致
2. 复用现有 ELF 装载器
3. 先验证文件系统闭环，再决定是否引入更真实的块设备层

当前内核路径是：

1. 构建阶段生成 `build/disk.img`
2. 使用 `ld -r -b binary` 把 `disk.img` 打进内核
3. 内核按 8.3 文件名从 root directory 查找 `TASKA.ELF` / `TASKB.ELF`
4. 取出文件内容后复用 `elf_load_image()`
5. 把装载后的程序加入任务队列

## 9. 工作包 6.6：文件系统边界与扩展说明

当前实现只覆盖 FAT32 的教学最小闭环：

1. 能创建镜像
2. 能展示 boot sector 关键元数据
3. 能向 root directory 放入两个 ELF 文件
4. 能由内核在运行时读取并装载这两个 ELF

后续可扩展方向：

1. 支持任意 cluster 链读取
2. 支持子目录与目录遍历
3. 支持写入普通文本文件
4. 支持在内核态引入真实 virtio-blk 或等价块设备驱动
5. 对比 NTFS、ext 系列、Lustre 等文件系统设计取舍

## 10. 下一步建议

完成当前 6.1 到 6.6 的最小闭环之后，建议下一步按以下顺序演进：

1. 把 root directory 解析独立成可测试模块。
2. 支持跨 cluster 文件读取。
3. 支持从镜像中读取共享配置或数据文件。
4. 评估是否引入真实块设备读盘路径。
