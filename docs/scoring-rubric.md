# SysToy 评分 Rubric

## 1. 评分原则

1. 以阶段完成度为主。
2. 以可运行、可说明、可复现为核心。
3. 基本分优先于进阶分和高端分。
4. 扩展内容不能掩盖基础功能缺失。

## 2. 评分维度

建议每阶段按以下维度评分：

1. 功能完成度
2. 工程规范性
3. 技术说明清晰度
4. 演示完整度

## 3. 通用分值建议

每阶段基础分建议按 10 分计：

| 维度 | 分值 | 说明 |
|---|---:|---|
| 功能完成度 | 5 | 是否完成阶段规定能力 |
| 工程规范性 | 2 | 目录、编码、命名、交付结构是否规范 |
| 技术说明 | 2 | 是否讲清楚实现思路、边界和局限 |
| 演示与答辩 | 1 | 是否可稳定展示并回答问题 |

## 4. 基本分判定

### 4.1 通过线

1. 核心功能必须可运行。
2. 阶段文档必须提交。
3. 演示必须可完成。
4. 无法启动或无法说明原理的项目，原则上不能通过该阶段。

### 4.2 不通过判定

存在以下任一情况时，可判定为不通过：

1. 关键功能无法运行。
2. 无法完成现场演示。
3. 无法解释实现路径和关键机制。
4. 文档缺失严重。

## 5. 进阶分建议

每阶段可加 1 分。进阶分要求：

1. 与当前阶段主题紧密相关。
2. 能清楚说明扩展目标与收益。
3. 至少有一种可观测结果。

示例：

1. Phase 2：对中断或系统调用做更清晰的抽象。
2. Phase 3：对内存管理策略做对比分析。
3. Phase 4：补充多种调度算法比较。
4. Phase 5：增加不同同步场景示例。
5. Phase 6：补充 FAT32 之外的文件系统分析。

## 6. 高端分建议

每阶段可加 1 分。高端分要求：

1. 明显超出基本要求。
2. 具备一定技术难度。
3. 有清晰边界，不能只是概念堆砌。

示例：

1. Phase 3：尝试按需分页原型。
2. Phase 4：更复杂的任务调度策略。
3. Phase 5：更完整的阻塞/唤醒机制。
4. Phase 6：文件系统性能优化或更复杂结构支持。

## 7. 各阶段最低关注点

### Phase 1

1. 环境能否跑通。
2. 工具链是否完整。
3. 是否提交自检结果。

### Phase 2

1. 是否成功从 MBR 进入内核。
2. 是否具备基础输出能力。
3. 是否能演示系统调用链路。

### Phase 3

1. 是否能加载并运行 ELF。
2. 是否解释清楚内存布局。

### Phase 4

1. 是否能调度两个及以上用户程序。
2. 是否能说明调度策略。

### Phase 5

1. 是否能展示共享内存和同步机制。
2. 是否能对比加锁与不加锁结果。

### Phase 6

1. 是否能识别和展示 FAT32 基本结构。
2. 是否能从文件系统加载 ELF。

## 8. 助教评分建议

1. 先看能否运行，再看实现是否优雅。
2. 先看是否完成主线，再看扩展亮点。
3. 评分记录中应保留扣分原因。
