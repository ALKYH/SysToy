# 系统实验统一项目产品需求文档（PRD）

## 1. 文档信息

- 文档名称：系统实验统一项目产品需求文档（PRD）
- 项目代号：SysToy
- 文档版本：v1.0
- 文档状态：Draft
- 编写日期：2026-05-26
- 适用对象：课程负责人、实验指导教师、项目助教、研发支持人员、学生项目组

## 2. 产品背景

在操作系统课程教学中，实验内容通常分散在多个相互独立的知识点中，学生虽然能够分别完成引导加载、内存管理、调度算法、同步机制与文件系统等局部任务，但难以建立“一个最小可运行操作系统如何逐步演进”的整体认知。这会带来以下问题：

1. 学生对实验之间的依赖关系理解不清，无法形成系统级工程视角。
2. 实验交付物割裂，难以沉淀为可复用的教学资产。
3. 教师在验收、评分和进度管理上缺乏统一标准。
4. 学生在环境搭建阶段耗时较长，影响后续核心内容推进。

基于上述问题，本项目拟以“统一项目、分阶段演进”的方式，设计并建设一套面向操作系统课程的实验型系统项目。项目以 Windows 10 为基础开发环境，以 WSL2、NASM、GCC、Make 和 QEMU 为核心工具链，以一个可逐步扩展的最小操作系统为主线，分六个阶段完成从环境搭建到文件系统支持的完整建设。

## 3. 产品愿景

构建一套可教学、可实施、可验收、可扩展的最小操作系统实验项目体系，使学生能够在统一工程框架下，逐步完成从 Bootloader 到内核基础能力、内存管理、调度、同步与文件系统的实现，并建立完整的系统软件工程认知。

## 4. 产品目标

### 4.1 业务目标

1. 建立统一的操作系统实验项目主线，替代碎片化实验安排。
2. 降低课程实施成本，提升实验项目的复用性与可维护性。
3. 形成可标准化执行的验收机制、评分机制与阶段交付机制。
4. 支持后续课程迭代、进阶扩展与优秀项目沉淀。

### 4.2 用户目标

1. 学生能够在统一环境中完成项目开发、调试、运行与展示。
2. 学生能够理解操作系统核心模块之间的依赖与演进逻辑。
3. 教师能够基于阶段成果进行过程管理与阶段验收。
4. 助教能够依据统一标准进行问题定位、指导与评审。

### 4.3 成功指标

1. 80% 以上学生团队能够在规定周期内完成前 4 个阶段的基本要求。
2. 60% 以上学生团队能够完成前 5 个阶段的基本要求。
3. 40% 以上学生团队能够完成第 6 阶段的基本要求。
4. 100% 的实验项目均具备统一交付物、统一演示口径和统一验收标准。
5. 环境搭建相关问题在课程支持工单中的占比低于 30%。

## 5. 产品范围

### 5.1 范围内

本项目覆盖以下六个阶段能力建设：

1. Windows 10 下实验开发环境建设。
2. Bootloader（MBR）与 Naive Kernel 构建。
3. 不考虑 Paging 的内存管理机制。
4. 基于时钟中断的用户程序调度。
5. 基于共享内存与 P/V 语义的同步机制。
6. 基于 FAT32 的文件系统认知、虚拟盘构建与 ELF 加载执行。

### 5.2 范围外

以下内容不属于本期 PRD 的必做范围：

1. 完整现代操作系统功能实现。
2. 图形界面系统。
3. 网络协议栈与分布式运行时。
4. 多核 SMP 完整支持。
5. 完整虚拟内存、按需分页和 64 位地址空间管理。
6. 完整文件系统优化、日志恢复与高可靠存储设计。

## 6. 目标用户与角色定义

### 6.1 目标用户

1. 本科阶段操作系统课程学生。
2. 需要组织系统实验教学的教师团队。
3. 负责实验支持、答疑与验收的助教团队。

### 6.2 角色定义

1. 产品负责人/课程负责人：定义阶段目标、验收规则与评分机制。
2. 项目经理/教学协调人：跟踪里程碑、进度风险与资源协调。
3. 技术负责人：统一技术路线、代码基线与实验规范。
4. 指导教师/助教：支持问题排查、阶段验收与质量把关。
5. 学生项目组：完成阶段开发、测试、汇报与交付。

## 7. 核心用户场景

### 场景 1：课程启动

课程负责人在学期开始前发布统一项目说明、阶段目标、代码基线和交付模板，各学生项目组基于统一主线开展实验。

### 场景 2：阶段推进

学生团队按阶段推进，先完成环境与引导，再逐步扩展内核能力。每个阶段均输出可运行结果与阶段文档，由教师或助教进行验收。

### 场景 3：过程评审

在每个里程碑节点，教师依据统一验收标准检查功能实现、演示结果、代码结构和项目说明材料，对团队进行评分与反馈。

### 场景 4：能力拓展

在完成基本要求后，学生可基于“进阶分”和“高端分”机制，探索更远但关联度较高的能力扩展，如 VM、内核定制、分布式文件系统等。

## 8. 产品总体方案

本项目采用“一个统一操作系统实验产品，六个版本阶段演进”的模式实施。各阶段不是相互独立的孤立实验，而是在前一阶段成果基础上持续演进。

总体架构演进路径如下：

1. Phase 1：构建统一开发与运行环境，保证基础工具链可用。
2. Phase 2：建立最小可启动系统，完成从 Bootloader 到 Naive Kernel 的引导和基础输出能力。
3. Phase 3：为内核增加基础内存装载与用户程序运行能力。
4. Phase 4：为多个用户程序提供基于时钟中断的调度执行能力。
5. Phase 5：在多程序运行基础上加入共享内存与同步机制。
6. Phase 6：引入虚拟盘与 FAT32 文件系统，使用户程序可从文件系统中加载并运行。

## 9. 功能需求

### 9.1 Phase 1：环境搭建

#### 9.1.1 目标

建立统一、稳定、可验证的实验开发环境，为后续所有阶段提供基础支撑。

#### 9.1.2 功能要求

1. 项目支持在 Windows 10 平台进行开发。
2. 项目具备 WSL2 环境支持能力。
3. 项目具备 NASM、GCC、Make 等编译工具链支持能力。
4. 项目具备 QEMU 运行与验证能力。
5. 学生能够基于给定代码完成编译、启动与结果验证。

#### 9.1.3 验收标准

1. 学生团队可在指定环境中完成基础工程编译。
2. QEMU 能够正常加载指定镜像并运行。
3. 团队能够提交环境自检结果与运行截图或演示记录。

### 9.2 Phase 2：Bootloader（MBR）+ Naive Kernel

#### 9.2.1 目标

构建最小可引导系统，实现从 MBR 到内核的基础启动链路，并建立中断封装与基础系统调用入口。

#### 9.2.2 功能要求

1. 能够修改并扩展 MBR 代码，使其被 QEMU 正常加载和运行。
2. 内核具备基础输出能力，支持类似 `Printf()` 的封装接口。
3. 内核具备中断封装能力。
4. 支持 CPU Switching 相关基础能力。
5. 实现用户态与内核态之间的基本切换机制。
6. 提供一个内核外部程序，用于展示 System Call 的调用过程。

#### 9.2.3 验收标准

1. 系统镜像能够从 Bootloader 正常进入内核。
2. 内核可通过封装接口输出稳定、可识别的调试信息。
3. 用户程序能够通过约定接口触发系统调用，并得到预期反馈。

### 9.3 Phase 3：内存管理（不考虑 Paging）

#### 9.3.1 目标

在不引入分页机制的前提下，实现基础内存管理与 ELF 用户程序装载执行能力。

#### 9.3.2 功能要求

1. 内核实现类似 Variable Partitioning 的基础内存管理思路。
2. 内核能够将编译好的 ELF 程序加载至内存。
3. 内核能够完成用户程序装载后的执行。
4. 在尚未引入文件系统的前提下，支持通过静态拷贝方式将用户程序放入用户空间并运行。

#### 9.3.3 验收标准

1. 至少一个 ELF 用户程序能够被内核加载并运行。
2. 内存分配与装载过程具备可说明性与可演示性。
3. 团队能够清晰说明当前方案为何不涉及 Paging，边界在哪里。

### 9.4 Phase 4：调度算法

#### 9.4.1 目标

在已有单用户程序执行能力基础上，引入基于时钟中断的多用户程序调度能力。

#### 9.4.2 功能要求

1. 除已有用户程序外，再构建至少一个额外用户程序。
2. 内核能够借助时钟中断提供调度机制。
3. 内核能够在多个用户程序之间进行切换与运行管理。
4. 调度策略至少实现一种可说明的基本算法。

#### 9.4.3 验收标准

1. 两个及以上用户程序可在统一内核中被调度运行。
2. 调度过程具备可观测输出。
3. 团队能够说明所选调度策略、切换依据及局限性。

### 9.5 Phase 5：Synchronization

#### 9.5.1 目标

在多程序调度基础上，实现共享内存与同步机制，使学生能够理解并演示竞争条件与锁语义。

#### 9.5.2 功能要求

1. 内核提供共享内存能力。
2. 内核提供仿真 P-V 操作的同步机制。
3. 至少构建两个用户程序共同访问共享资源。
4. 示例场景能够体现“并发访问导致的数据不一致”与“加入同步后的结果正确性”。

#### 9.5.3 验收标准

1. 两个用户程序能够基于共享内存进行交互。
2. 程序运行结果能够体现同步机制有效性。
3. 团队能够解释锁机制、临界区与竞争条件。

### 9.6 Phase 6：File System

#### 9.6.1 目标

引入虚拟盘与 FAT32 文件系统，使用户程序能够作为文件被保存、识别、加载并执行。

#### 9.6.2 功能要求

1. 选择一个文件系统，本项目基础要求为 FAT32。
2. 能够创建虚拟盘并完成 FAT32 格式化。
3. 能够展示必要的文件系统信息，如头信息与内部组织结构。
4. 能够将 ELF 程序保存到虚拟盘中。
5. 改造后的内核能够从 FAT32 中加载 ELF 程序并执行。

#### 9.6.3 验收标准

1. 团队能够展示虚拟盘创建、格式化与结构识别结果。
2. 团队能够展示 ELF 文件写入虚拟盘后的加载执行结果。
3. 团队能够说明 FAT32 的核心设计要点与当前实现边界。

## 10. 非功能需求

### 10.1 可用性

1. 所有阶段应基于统一工程基线推进。
2. 阶段成果应具备可重复运行能力。
3. 运行与演示方式应尽量标准化。

### 10.2 可维护性

1. 项目代码应按阶段持续沉淀，不鼓励每阶段重新起项目。
2. 每阶段交付物应包含必要说明，以支持教师与助教快速定位问题。
3. 项目应支持后续学期复用与版本迭代。

### 10.3 可观测性

1. 关键阶段必须有可展示结果，如输出日志、运行结果、演示视频或现场演示。
2. 关键模块应具备基础调试输出能力。

### 10.4 一致性

1. 所有团队采用统一命名、统一交付结构、统一验收口径。
2. 所有文档、代码与说明材料统一使用 UTF-8 编码。

## 11. 里程碑规划

### 11.1 总体节奏

建议按照“基础设施优先、内核主线递进、阶段性交付验收”的方式推进。

### 11.2 里程碑划分

1. M1：完成环境搭建与基线运行验证。
2. M2：完成最小可启动系统与系统调用演示。
3. M3：完成 ELF 装载与基础内存管理。
4. M4：完成双用户程序调度运行。
5. M5：完成共享内存与同步演示。
6. M6：完成 FAT32 虚拟盘、文件读取与 ELF 装载运行。

### 11.3 阶段关系

1. Phase 1 是全项目基础前提。
2. Phase 2 是后续所有内核能力扩展的起点。
3. Phase 3 为 Phase 4 和 Phase 5 提供用户程序运行基础。
4. Phase 6 是对前述用户程序运行机制的进一步外部化与工程化。

## 12. 交付物要求

每个阶段应至少包含以下交付物：

1. 可运行代码或镜像。
2. 阶段说明文档。
3. 运行结果证明材料。
4. 问题记录与解决说明。
5. 阶段汇报材料或演示说明。

全项目最终交付物应包括：

1. 统一项目源码。
2. 最终系统说明文档。
3. 六阶段演进说明。
4. 最终演示材料。
5. 扩展功能说明材料（如有）。

## 13. 验收方案

### 13.1 验收原则

1. 以“可运行、可说明、可复现”为核心。
2. 以阶段验收为主，期末总验收为辅。
3. 基本功能完成度优先于复杂扩展。

### 13.2 验收维度

1. 功能完成度。
2. 系统运行稳定性。
3. 代码组织与工程规范性。
4. 技术说明清晰度。
5. 演示完整度。

### 13.3 验收方式

1. 阶段现场演示。
2. 文档审查。
3. 代码审查。
4. 问答式答辩。

## 14. 评分机制建议

### 14.1 基本分

完成各阶段基本要求即可获得该阶段基本分。基本分聚焦于指定功能实现与阶段验收通过。

### 14.2 进阶分

每个项目可增加 1 分。进阶分定义为：在完成基本要求的前提下，增加与当前主题“距离不太远”的知识扩展或功能扩展。

示例包括：

1. 在文件系统阶段，扩展介绍 NTFS、Linux 文件系统或 Lustre 等相关概念与设计。
2. 在 FAT32 基础上尝试增加索引、查询优化等增强能力。
3. 在调度阶段对多种经典调度算法进行对比说明与演示。

### 14.3 高端分

每个项目可增加 1 分。高端分定义为：在基本分与进阶分之外，增加与主题“距离更远但仍具技术关联”的深入实现或系统拓展。

示例包括：

1. 在内存管理阶段实现 on-demand paging 的 VM 模块。
2. 探索 64 位虚拟内存空间支持。
3. 将图形界面、安全机制、并行计算、网络能力等集成进内核。
4. 探索分布式文件系统或大数据相关系统能力，如 HDFS、Zookeeper、K8s 等周边方向。

## 15. 风险与依赖

### 15.1 关键依赖

1. Windows 10 基础环境可用。
2. WSL2 与相关工具链安装条件满足。
3. 教师团队具备统一基线代码和统一指导口径。
4. 学生具备基础 C 语言与一定汇编阅读能力。

### 15.2 主要风险

1. 环境搭建失败导致课程初期进度普遍滞后。
2. 学生对引导加载与中断机制理解不足，Phase 2 成为主要阻塞点。
3. 项目阶段依赖紧密，前置阶段未完成会影响后续全部进度。
4. 文档与代码缺乏统一规范，会增加验收与支持成本。
5. 进阶分与高端分范围不清，可能造成评分尺度不一致。

### 15.3 风险应对

1. 在正式启动前提供统一基线与预检查机制。
2. 将 Phase 2 作为重点辅导阶段，设置专项答疑。
3. 阶段验收不过时，必须先完成整改再进入下一阶段。
4. 提前明确评分 rubric 与扩展边界。

## 16. 资源需求

### 16.1 人力资源

1. 课程负责人 1 名。
2. 技术负责人 1 名。
3. 助教若干名。
4. 学生按组参与。

### 16.2 技术资源

1. 统一代码仓库。
2. 统一实验文档模板。
3. 统一演示与验收模板。
4. 统一问题登记与答疑渠道。

## 17. 项目治理机制

### 17.1 例会机制

1. 建议按周进行项目进展同步。
2. 每阶段结束前进行阶段评审。

### 17.2 变更管理

1. 基线需求原则上在学期中不做大改。
2. 如需变更验收标准，应由课程负责人统一发布。
3. 扩展内容不应影响基本分主线交付。

### 17.3 质量管理

1. 采用阶段验收卡点机制保障质量。
2. 对重复性问题形成 FAQ 或指导清单。
3. 对优秀项目进行归档，沉淀为下一轮课程资产。

## 18. 结项标准

当以下条件满足时，视为本统一项目完成：

1. 六个阶段均已具备明确需求说明与验收标准。
2. 学生团队能够基于统一主线完成阶段性实现与展示。
3. 教师与助教能够基于统一交付物开展验收与评分。
4. 项目形成可复用的代码、文档、评分与答疑资产。

## 19. 附录：术语说明

1. MBR：Master Boot Record，主引导记录。
2. Bootloader：引导加载程序。
3. Kernel：操作系统内核。
4. System Call：系统调用，用户态请求内核服务的接口。
5. ELF：Executable and Linkable Format，可执行与可链接格式。
6. Variable Partitioning：可变分区式内存管理思路。
7. Synchronization：同步机制，用于约束并发访问顺序与互斥关系。
8. FAT32：一种经典文件系统格式。
