# SysToy LLM 友好型工作拆解与工期规划

> 面向 Agent/LLM 的使用方式：先按阶段切分，再按工作包切分，最后把单个工作包拆成一次会话内可独立完成、可验证、可交付的小任务。

**目标：** 将 SysToy 课程项目按 LLM 友好的方式拆解为低耦合、可快速验证、上下文压力可控的工作包，并给出建议工期。  
**主线：** 以 `Phase 0` 到 `Phase 6` 为主线，每个阶段定义目标、依赖、交付物、验收标准和推荐的 LLM 子任务。  
**技术栈：** Windows 10, WSL2, NASM, GCC, Make, QEMU, x86/RISC-V 教学内核, ELF, FAT32, Markdown 文档体系

---

## 1. 拆解原则

### 1.1 什么叫“LLM 友好”

一个任务如果满足以下条件，就适合交给 LLM 或 Agent 单次执行：

1. 目标单一，只解决一个明确问题。
2. 作用范围小，涉及文件最好控制在 3 到 6 个核心文件内。
3. 输入明确，前置资料可以一次给全。
4. 输出明确，能清楚说明要产出代码、文档、脚本还是验证结果。
5. 验收明确，完成后能立即判断“是否完成”。
6. 不依赖超长隐式上下文，后续任务可以通过文档继续接力。

### 1.2 不适合直接交给单次 LLM 的任务

以下任务不应作为一个工作包整体下发：

1. “把整个内核做完”这类范围过大的目标。
2. 同时跨越 Bootloader、内存、调度、文件系统的复合修改。
3. 没有可执行验收口径，只能靠主观印象判断的任务。
4. 严重依赖长期上下文且无法通过输入材料补齐的信息型任务。

### 1.3 建议的任务时长

建议按以下三档组织工作：

1. 微任务：0.5 天内，适合单文件修复、小文档、小脚本、小验证。
2. 标准任务：1 天，适合一个明确功能点或一个独立子模块。
3. 复合任务：2 天，适合“小闭环”，例如“新能力 + 演示样例 + 说明文档”。

不建议把单次 LLM 任务设计成超过 2 天的人类工作量，否则返工和上下文失真风险会明显上升。

## 2. 总体工期建议

### 2.1 建议总工期

作为完整课程项目主线，建议总工期控制在 12 到 16 周。

### 2.2 各阶段建议工期

1. Phase 0：项目基线与治理准备，1 周。
2. Phase 1：环境搭建，1 周。
3. Phase 2：Bootloader 与 Naive Kernel，2 到 3 周。
4. Phase 3：内存管理与 ELF 装载，2 周。
5. Phase 4：调度算法，2 周。
6. Phase 5：Synchronization，2 周。
7. Phase 6：File System，2 到 3 周。
8. 缓冲与验收收尾，1 周。

### 2.3 工作包规模建议

建议总共拆成 30 到 45 个标准化工作包。每个工作包尽量保持：

1. 1 个明确目标。
2. 1 套主要产出。
3. 1 套清晰验收标准。
4. 不超过 2 个强前置依赖。

## 11. 推荐的 LLM 任务模板

建议后续所有 LLM/Agent 子任务统一使用下面结构：

1. 任务名称
2. 背景说明
3. 输入材料
4. 明确目标
5. 非目标
6. 预期输出
7. 验收标准
8. 约束条件
9. 涉及文件范围
10. 建议先读文件

示例：

1. 任务名称：为 Phase 2 增加最小 System Call 入口
2. 背景说明：当前 Bootloader 和 Kernel 已能启动，但用户态程序还不能稳定进入内核服务
3. 输入材料：Phase 2 设计文档、当前中断处理文件、用户程序样例
4. 明确目标：新增最小 system call 入口，支持一个演示调用
5. 非目标：不实现完整 syscall 表，不设计权限模型
6. 预期输出：代码变更、调试输出、演示说明
7. 验收标准：用户程序能调用内核并看到预期输出

## 12. 推荐排期方式

### 12.1 课程推进视角

建议以“周”为外层节奏，以“工作包”为内层执行单元：

1. 每周只推进 2 到 4 个工作包。
2. 每个工作包再拆成 1 到 3 个 LLM 子任务。
3. 每周至少预留 20% 缓冲处理环境、返工和验收问题。

### 12.2 Agent 协作视角

建议采用以下节奏：

1. 一个 Agent 只拿一个标准任务。
2. 设计任务与实现任务分开交付。
3. 代码任务与文档任务尽量拆成前后两个连续任务。
4. 涉及跨阶段依赖的任务必须附带上游摘要。

### 12.3 最佳任务粒度建议

最推荐的粒度是：

1. 一次只变更一个机制。
2. 一次只新增一个演示样例。
3. 一次只补一种文档。
4. 一次只解决一个阻塞问题。
